# 用語集（glossary）

howWhatWhy で使う言葉の定義。曖昧さを避けるための共有辞書。
設計の経緯は [[adr]]、運用ルールは [[rule]] を参照。

## how（事実）
実際にやった行動と、その結果。解釈を含まない。ソフト開発のコードと実行結果に相当。

## what（達成条件）
何ができたら成功かの定義。ソフト開発のテストに相当。

## why（判断 / ADR）
なぜその手段を選んだ／選ばなかったかの意思決定の記録。Nygard 版 ADR。

## learn（知見）
プロジェクト固有の学び。次の2種を含む:
- **解釈**: 特定結果の読み解き（「テスト X が落ちたのは Y のため」）。後ろ向き。
- **教訓**: 一般化した学び（「Y を避けるには Z せよ」）。前向き。

## 知見（learn の1エントリ）
具体的な観察・例。1つの出来事から得た学び。これが昇格の素材になる。

## 原則（principle）
複数の知見が共通して指している抽象。`principles/` に置く。
1ファイル1原則で、原則を見出しに、それを支える具体例（知見）を並べる。

## 昇格（promotion）
バラバラの知見が同じ原則を指していると見抜き、`principles/` に原則として書き出す行為。
チェーン: `learn` → `principles` →（将来）`zettelkasten`。
**判定基準**（何個で束ねるか）と **起動**（いつ棚卸しするか）は別物。

## 判定基準（昇格の閾値）
昇格に値するかの判断。おおむね2〜3例が同じ原則を指したら原則化を検討（目安）。
1例でも明らかに普遍なら可、弱ければ保留。最終は AI 提案＋人間の承認。

## 棚卸し（review / サボらないループ）
昇格を起こすための定期的な見直し。二層トリガーで回す:
- **層1（日和見・任意）**: 人間が「ついで」に行う。スキップ可。
- **層2（バックストップ・自動）**: 毎週、エージェントが自動で行う。人間の起動を要しない。

## push 型 / pull 型トリガー
- **pull 型（人間が始める）**: あなたの意志・記憶が必要。怠けると永久に始まらない。
- **push 型（自動で始まる）**: 時計・cron・エージェントが勝手に発火する。あなたが何もしなくても動く。
  バックストップ（棚卸しの層2）は必ず push 型にする。

## MOC（Map of Content / 地図ノート）
リンクを束ねる普通のノート。手で書く目次・ダッシュボード。
フォルダと違い、1つのノートが複数の MOC に属せる（多対多）。`maps/` に置く。
カテゴリ（frontend, 設計…）はフォルダではなく MOC／タグで表す。

## 運用層（operational layer）
HWW（記録・過去）とは別の、前向きの層。プロジェクトの状態・実行計画・促しを扱う。
vault 内のメタdataから生成するビューとして持つ（別ツールにしない。[[adr]] ADR-0011）。

## 実行計画（regimen）
目的地（what）に到達するための道順: 頻度・一日量（dose）・タイミング。
what（達成条件）とは別物。型を混ぜないため what.md には書かない。

## status（プロジェクトの状態）
プロジェクトの運用状態（active / paused / done など）。運用層が追跡する。
具体的な語彙と遷移は運用ルールで定める。

## nudge（促し）
「やれ」と突きつける push。ノートは push しないので、
外部リマインダーや週次エージェントが担う（[[adr]] の push 型, ADR-0008）。
