# jotatsu 運用ルール

## コンセプト

ドキュメントは全て文字情報だが、その「**認識のタイプ**」がごちゃ混ぜになると、
何が事実で、何が解釈で、何が達成条件で、何が経緯なのかが区別できず、
プロジェクト達成の妨げになる。

whatWhyLearn は、ノートを**認識のタイプで物理的に隔離する**ことで、
書く時に迷わず、目標達成に近づくための運用ルールである。

着想元は、開発者としての以下の思想:

1. テストはコードの理想の動作を表す唯一のドキュメントである（what）。
2. ADR は現在のコードや技術選定の意思決定を記録する（why）。
3. 実務のコードは lint・規約・harness のような規範知にも支えられている。
   「学んだこと・アンチパターン・うまくやる方法」をまとめる場所が、
   人間のメモには最も重要である（learn）。
4. 動く実物（コード）自体は native な住処に在り、vault は持たない。生の作業記録は `note` に置く
   （how＝事実は「実物（native）」と「生ノート（note）」に分かれた。[[adr]] ADR-0015〜0017）。

## 3つの認識タイプ（核）＋ 生ノート

| ファイル | 認識タイプ | ソフトウェア開発の対応物 | 問い |
|---|---|---|---|
| `what.md`  | 達成条件 | テスト                      | 何ができたら成功か |
| `why.md`   | 判断     | ADR                         | なぜこの手段を選んだ／選ばなかったか |
| `learn.md` | 知見（craft） | lint・規約・harness・ポストモーテム | どうすればうまくいくか／何を避けるか |
| `note.md`  | 生の原料（核外） | git log・作業メモ | 何をやって何が起きたか（＋今こうしよう） |

核（what/why/learn）は「現在・確定した情報」だけ（[[adr]] ADR-0016）。`note.md` は生の自由記述で、
分離は「書く時」でなく「蒸留する時」に効かせる（ADR-0017）。

## 2階建て構造

```
jotatsu/                     ← vault ルート
├── principles/              ← 【全プロジェクト横断の一般法則】learn から昇格した原則（フラット・リンク）
├── maps/                    ← MOC（分類・案内の地図ノート。カテゴリの代わり。多対多）
├── template/                ← what/why/learn/note の雛形
└── projects/                ← プロジェクトのフラットな一覧（カテゴリでネストしない）
    └── <project>/
        ├── what.md          ← 達成条件
        ├── why.md           ← 判断
        ├── learn.md         ← 知見（craft・このプロジェクト固有）
        └── note.md          ← 生の作業ノート（核外・自由記述）
```

- **project 層**: 各プロジェクトの what/why/learn（＋生 note）。時限的な記録。寿命あり。
- **principles 層**: プロジェクトを超えて再利用する普遍知。寿命なし。原子的・リンクで接続。

## ルール

1. **1プロジェクト = 1フォルダ**。中身は what/why/learn の3ファイル＋生 `note.md`＋運用メタ `index.md`（表紙・frontmatter）。プロジェクトは `projects/` にフラットに置く。
2. `note` は**生の作業ノート**（やった事実・「今こうしよう」・気づきの種を混ぜて書く）。分類は書く時でなく蒸留時に。実物（コード等）は外部に置き link。
3. `what` は**成功の定義**（達成条件）。テストに相当する。
4. `why` は**意思決定**の記録。採用（Why）／不採用（Why not）の両面を書く。ADR に相当。
5. `learn` は**知見（仕事の進め方・craft）**。次の両方を入れてよい:
   - 特定結果の解釈（例: 「テスト X が落ちたのは Y のせい」）
   - 一般化した教訓（例: 「Y を避けるには常に Z せよ」）
   - 技術/ドメインそのものの知識（例「net/http の書き方」）は learn でなく zettelkasten へ（[[adr]] ADR-0017）。
6. **昇格の判定基準**: 同じ原則を指す例がおおむね **2〜3個** 揃ったら原則化を検討する（目安）。
   1例でも明らかに普遍なら昇格可、弱ければ保留。最終は昇格案を AI が出し、あなたが承認／却下する。
7. `principles/` は蒸留済み・普遍・再利用される原則。**1ファイル1原則**とし、
   原則を見出しに、それを支える具体例（知見）を下に並べる。`[[リンク]]` で繋ぐ。

## 棚卸し（サボらないループ）

昇格を実際に起こすための定期的な見直し。**人間の意志に依存させない**のがキモ。
判定基準（ルール6）とは別の、「いつ棚卸しが起きるか」の仕組み。

- **層1（日和見・任意）**: プロジェクト完了時などに `learn.md` をざっと見て、動かせる原則は自分で `principles/` へ。スキップ可。
- **層2（バックストップ・自動・絶対）**: 毎週、棚卸しエージェントが全 `learn.md` を読み、昇格案を作ってあなたに承認／却下を求める。**あなたは何も始めない**（push 型）。
- 層1をサボっても、層2が必ず拾う。これがメモを続かせる仕組み。

## 運用層（プロジェクト管理 / トラッカー）

「今アクティブか・抱えすぎてないか・どの頻度でやるか」を扱う前向きの層。
真実の源は各プロジェクトの `index.md` frontmatter、トラッカーは `dashboard.md` の Dataview が
`projects/` を走査して自動生成する（[[adr]] ADR-0011, ADR-0012）。

- **状態(status)**: `backlog` → `active` → `done`、随時 `paused`、見切りは `dropped`。**dropped を正直に使う。**
- **WIP 上限 = 5**: 同時 `active` は最大5。6つ目を始めるには、まず1つを done/paused/dropped にする。
- **停滞判定 = 14日**: `active` かつ last-touched から14日超 → 停滞。週次レビューで 再開／paused／dropped を強制決定。
- **frontmatter**: `status / started / last-touched / cadence / dose / tags`（cadence・dose は稽古系のみ）。
- **促し(nudge)**: ダッシュボードは pull。停滞・WIP超過の push は週次エージェント、日次の稽古は外部リマインダー。

## 捕捉（inbox）と整理を分ける

疲れているときに全ルールを守るのは無理。だから捕捉と整理を分ける（[[adr]] ADR-0013）。

- **捕捉**: `inbox.md` に時刻付きで1行放り込む。**ルールなし。** what/why/learn を考えない。
- **整理**: 後で各行を project の note/learn へ振り分け／新 project 化／principle 化／**削除**。
- **inbox は通り抜ける場所**。溜めない。**7日**超の未処理は処理 or 削除（月曜の点検が push で促す）。
- 走り書きは消耗品。解読不能なら捨てるのが正解。

## ADR（why）の運用

- **テンプレート**: Nygard 版（`Status` / `Date` / `Context` / `Decision` / `Consequences`）。
- **最小スタート**: 各プロジェクトは `why.md` 1枚に、`# ADR-NNNN: タイトル` を inline で並べる。
- **昇格（分割）**: 決定が増えて1ファイルが痛くなったら、`why/0001-*.md` の**1決定1ファイル**へ割る。
- **不変・追記専用**: 過去の ADR は書き換えない。決定が変わったら、新しい ADR で
  `Status: Superseded by [[ADR-NNNN]]` として上書きする。

## 階層が深くなる問題への対策（課題2）

根本原因: フォルダ（木）に「包含」と「分類」の2つの仕事をさせると破綻する。
木は1軸・1対多しか表せず、分類は多対多・無限に深くなる（詳細は [[adr]] ADR-0010）。

道具を仕事で分ける:
- **包含 → フォルダ（1階層）**: プロジェクトは `projects/` にフラットに並べる。カテゴリでネストしない。
- **分類 → タグ／MOC（グラフ）**: frontend・設計・きれいなコード等のカテゴリは絶対にフォルダにしない。
  frontmatter のタグと MOC（リンクを束ねる地図ノート, `maps/`）で表す。多対多OK・重複なし。
- **粒度の決定**: 「達成条件（what）があるか?」→ ある=プロジェクト=フォルダ／ない=カテゴリ=タグ・MOC。

加えて、一般知は「下に埋もれる」のではなく `principles/` へ「**上に浮上**」する。
深さで整理せず、昇格とグラフ（タグ／MOC）で整理する。これがネスト爆発を防ぐ。

## principles/ への昇格と、その先（課題3は一旦保留）

- 知識の昇格チェーン: `learn`（プロジェクト固有）→ **棚卸しで束ねて** `principles/`（全プロジェクト横断の一般法則）。
- `principles/` からさらに普遍的・時間に依らない知見が出てきたら、
  jotatsu と**同じ階層の兄弟ディレクトリ `zettelkasten/`** へ移す。
- ただし `zettelkasten/` の設計は今はしない（スコープ外）。
  当初の課題3「Zettelkasten と whatWhyLearn の使い分け」は、必要になった時点で別途設計する。

## 未決（運用しながら修正してよい）

- `learn`（craft）と `note`（生の事実）の境界は、運用しながら調整する。
- `principles/` の最初の作成タイミング、ファイル命名規約。
- 三つ組の単位を「課題単位」に細分化する必要が出るかどうか。

詳しい採用／不採用の経緯は [[adr]] を参照。
