# howWhatWhy ノート運用ルール

## コンセプト

ドキュメントは全て文字情報だが、その「**認識のタイプ**」がごちゃ混ぜになると、
何が事実で、何が解釈で、何が達成条件で、何が経緯なのかが区別できず、
プロジェクト達成の妨げになる。

howWhatWhy は、ノートを**認識のタイプで物理的に隔離する**ことで、
書く時に迷わず、目標達成に近づくための運用ルールである。

着想元は、開発者としての以下の思想:

1. コードは実際に動作するものであり、how（事実）を表す。
2. テストはコードの理想の動作を表す唯一のドキュメントである（what）。
3. ADR は現在のコードや技術選定の意思決定を記録する（why）。
4. ただし実務のコードは lint・規約・harness のような規範知にも支えられている。
   「学んだこと・アンチパターン・うまくやる方法」をまとめる場所が、
   人間のメモには最も重要である（learn）。

## 4つの認識タイプ

| ファイル | 認識タイプ | ソフトウェア開発の対応物 | 問い |
|---|---|---|---|
| `how.md`   | 事実     | コードとその実行結果        | 何をやって、何が起きたか |
| `what.md`  | 達成条件 | テスト                      | 何ができたら成功か |
| `why.md`   | 判断     | ADR                         | なぜこの手段を選んだ／選ばなかったか |
| `learn.md` | 知見     | lint・規約・harness・ポストモーテム | 何を学んだか／どうすればうまくいくか／何を避けるか |

## 2階建て構造

```
howWhatWhy/                  ← vault ルート
├── principles/              ← 【全プロジェクト横断の一般法則】howWhatWhy 内の最上位の知識層
│   └── *.md                 ←   蒸留済み・普遍・再利用される知見。1ファイル1法則。[[リンク]]で繋ぐ
├── template/                ← how/what/why/learn の雛形
└── <プロジェクト名>/
    ├── how.md               ← 事実
    ├── what.md              ← 達成条件
    ├── why.md               ← 判断
    └── learn.md             ← 知見（このプロジェクト固有）
```

- **project 層**: 各プロジェクトの how/what/why/learn。時限的な実行記録。寿命あり。
- **principles 層**: プロジェクトを超えて再利用する普遍知。寿命なし。原子的・リンクで接続。

## ルール

1. **1プロジェクト = 1フォルダ = how/what/why/learn の4ファイル。**
2. `how` は純粋な**事実**（やった行動と、その結果）。解釈を混ぜない。
3. `what` は**成功の定義**（達成条件）。テストに相当する。
4. `why` は**意思決定**の記録。採用（Why）／不採用（Why not）の両面を書く。ADR に相当。
5. `learn` は**知見**。次の両方を入れてよい:
   - 特定結果の解釈（例: 「テスト X が落ちたのは Y のせい」）
   - 一般化した教訓（例: 「Y を避けるには常に Z せよ」）
6. **昇格ルール**: `learn.md` に同じ知見が **3回** 出てきたら、`principles/` に昇格させる。
   プロジェクトを超えて成り立つ一般法則とみなす。
7. `principles/` は蒸留済み・普遍・再利用される知見。**1ファイル1法則**とし、
   `[[リンク]]` で繋ぐ（原子的に運用する）。

## ADR（why）の運用

- **テンプレート**: Nygard 版（`Status` / `Date` / `Context` / `Decision` / `Consequences`）。
- **最小スタート**: 各プロジェクトは `why.md` 1枚に、`# ADR-NNNN: タイトル` を inline で並べる。
- **昇格（分割）**: 決定が増えて1ファイルが痛くなったら、`why/0001-*.md` の**1決定1ファイル**へ割る。
- **不変・追記専用**: 過去の ADR は書き換えない。決定が変わったら、新しい ADR で
  `Status: Superseded by [[ADR-NNNN]]` として上書きする。

## 階層が深くなる問題への対策（課題2）

- 一般知は「下に埋もれる」のではなく「**上に浮上**」する。
- サブ課題はフォルダをネストせず、各ファイル内のセクションで扱う。
- **深さで整理せず、昇格で整理する**。これがネスト爆発を防ぐ。

## principles/ への昇格と、その先（課題3は一旦保留）

- 知識の昇格チェーン: `learn`（プロジェクト固有）→ **3回出たら** `principles/`（全プロジェクト横断の一般法則）。
- `principles/` からさらに普遍的・時間に依らない知見が出てきたら、
  howWhatWhy と**同じ階層の兄弟ディレクトリ `zettelkasten/`** へ移す。
- ただし `zettelkasten/` の設計は今はしない（スコープ外）。
  当初の課題3「Zettelkasten と howWhatWhy の使い分け」は、必要になった時点で別途設計する。

## 未決（運用しながら修正してよい）

- `learn` と `how` の結果欄の境界は、運用しながら調整する。
- `principles/` の最初の作成タイミング、ファイル命名規約。
- 三つ組の単位を「課題単位」に細分化する必要が出るかどうか。

詳しい採用／不採用の経緯は [[adr]] を参照。
