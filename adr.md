# ADR: jotatsu の設計判断

この設計（jotatsu というシステムそのものの設計）について、
ユーザーと Claude の対話で採用・不採用を決めた経緯を記録する。
形式は Nygard 版（Context / Decision / Consequences）の最小スタート。
運用ルールの要約は [[rule]] を参照。

決定が増えて1ファイルが痛くなったら、`adr/0001-*.md` のように
1決定1ファイルへ分割する（[[rule]] の「ADR の運用」を参照）。

---

# ADR-0001: 認識のタイプでノートを物理的に隔離する（howWhatWhyLearn を採用）

- Status: Superseded by [[ADR-0015]]（「隔離」の原則は継承。バケツ構成を how→log に変更）
- Date: 2026-06-28

## Context

Zettelkasten のようなものでメモを取ってきたが、事実／解釈／達成条件／経緯が
一緒くたになり、プロジェクト達成に不向きだと感じた。
Zettelkasten は意図的にフラットでアイデア・資料の保存向きであり、
プロジェクトトラッキング向きではない。文脈を区切れず凝縮度が足りない。

## Decision

ドキュメントを how（事実）／what（達成条件）／why（判断）で物理的に隔離する。
ソフトウェア開発の コード／テスト／ADR の対応に着想を得た。

## Consequences

- 書く時に「これは事実か、解釈か、条件か、経緯か」が明確になり、迷いが減る。
- Why not（棄却案）: Zettelkasten 単独で継続 → 文脈を区切れず凝縮度不足のため棄却。

---

# ADR-0002: 4つ目のバケツ `learn`（知見）を追加

- Status: Superseded by [[ADR-0015]]（learn は継承。バケツは how/what/why/learn → what/why/learn/log に再編）
- Date: 2026-06-28

## Context

how/what/why の3バケツに対し、当初分離したかった認識は4種類あった
（事実／解釈／達成条件／経緯）。「解釈」の居場所が無かった。
さらに、実務のコードも lint・規約・harness のような規範知に支えられており、
「学んだこと・アンチパターンの防止・どうすればうまくいくか」をまとめる場所が、
人間のメモには最も重要だと判明した。

## Decision

`learn.md` を追加する。how/what/why の3分割の美しさは犠牲にするが、
当初の4分類に忠実にする。

## Consequences

- 「最も重要な知見」を捨てずに済む。
- Why not（検討した代替案と不採用理由）:
  - why に統合 → 教訓は判断とは独立に再利用されるため不適。
  - how 内で明示分離 → 事実と解釈を混ぜると隔離の意図が薄れる。
  - すべて別所に逃がす → 未蒸留の知見で溢れる。
    一旦 `learn` に溜め、蒸留後に昇格する案（ADR-0004）を採用。

# ADR-0003: `learn.md` は「解釈」と「教訓」を両方持つ

- Status: Accepted
- Date: 2026-06-28

## Context

「特定結果の解釈（なぜ落ちたか）」と「一般化した教訓（こうすべき）」は
本来別物である（後ろ向き／前向き、寿命も異なる）。これを分けるべきか。

## Decision

同じ `learn.md` に両方入れる。書く時に迷わないことを優先する。
問題が出たら後で分離する。

## Consequences

- 運用開始時の境界判断コストが下がる。
- Why not: 解釈は how の結果欄・教訓は learn と最初から分ける案は、
  境界判断のコストが高く迷いを生むため見送り。

# ADR-0004: 知見の昇格ルール = 同じ知見が3回出たら `principles/` へ

- Status: Superseded by [[ADR-0007]]（棚卸し方式へ。「3回」は厳密閾値から目安へ格下げ）
- Date: 2026-06-28

## Context

`learn` → `principles` への昇格トリガーを決めないと、
`principles` が育たないか、逆に未成熟な知見で溢れる。

## Decision

同じ知見が3回出てきたら `principles/` に昇格させる。

## Consequences

- 昇格基準が明確になり、`principles` が一定の品質で育つ。
- Why not: 手動・気づいたら昇格 → 基準が無く属人的になり育たない懸念で棄却。

# ADR-0005: principles/ を最上位とし、Zettelkasten は一旦スコープ外にする

- Status: Accepted
- Date: 2026-06-28

## Context

当初は Zettelkasten を jotatsu 内に「2階建てで同居」させる案を検討した。
しかし設計を単純に保つため、Zettelkasten は一旦スコープ外とする。

## Decision

`principles/` を jotatsu 内の「全プロジェクト横断の一般法則」の最上位の知識層とする。
将来 `principles/` の中から、プロジェクトを超えてさらに普遍的・時間に依らない知見が出てきたら、
jotatsu と**同じ階層（兄弟ディレクトリ）の `zettelkasten/`** へ移す。
ただし `zettelkasten/` の設計は今はしない。

## Consequences

- 設計が単純になる。昇格チェーンは `learn` → `principles`（→ 将来 `zettelkasten`）。
- 一般知が深い階層に埋もれず「上に浮上」するため、階層の深さ問題（課題2）は引き続き緩和される。
- 当初の課題3「Zettelkasten との使い分け」は一旦保留。必要になった時点で別途設計する。
- Why not: Zettelkasten を jotatsu 内に同居させる案 → 設計が複雑になるため見送り。

# ADR-0006: ADR は Nygard 版・1ファイル1決定を標準とし、最小から始める

- Status: Accepted
- Date: 2026-06-28

## Context

ADR を1ファイルにまとめるか、1決定1ファイルにするか。
また、最初に採用するテンプレートをどれにするか。

## Decision

テンプレートは Nygard 版（Context / Decision / Consequences + Status / Date）を採用する。
各プロジェクトは `why.md` 1枚に inline で複数決定を書いて始め（最小スタート）、
決定が増えて痛くなったら `why/0001-*.md` の1決定1ファイルへ昇格する。

## Consequences

- 不変・追記専用・隔離・安定参照という ADR の利点を、必要になった時点で得られる。
- 「迷わない運用優先、問題が出たら後で分離」（ADR-0003）と一貫する。
- Why not: 最初から1決定1ファイルに分割 → 小規模では過剰で、運用開始の摩擦が大きいため見送り。

# ADR-0007: 昇格の判定基準 =「3回ちょうど」ではなく「おおむね2〜3例の目安＋判断」

- Status: Accepted
- Date: 2026-06-28

## Context

ADR-0004 は「同じ知見が3回出たら昇格」と厳密な回数で決めていた。
しかし「3回ちょうど」という閾値は硬すぎる。1例でも明らかに普遍的な原則はあるし、
逆に3例あっても偶然の一致で原則化に値しないこともある。

## Decision

昇格の判定を、回数の厳密閾値から**目安＋判断**に変える。

- 同じ原則を指す例が**おおむね2〜3個**揃ったら、原則化を検討する（目安）。
- ただし**1例でも明らかに普遍**なら昇格してよい。
- 例が弱い・偶然かもしれないものは**保留**（次の棚卸しに持ち越す）。
- 最終判断は AI が昇格案を出し、**人間が承認／却下**する。

## Consequences

- 「3回」に縛られず、原則の質で判断できる。
- ADR-0004 を Supersede する。
- これは「何個で束ねるか」という**判定**の決定であり、「いつ棚卸しするか」という
  起動の決定（ADR-0008）とは独立。
- Why not: 厳密な回数閾値 → 質を無視して機械的になり、良い1例を逃し悪い3例を拾う。

# ADR-0008: サボらないループ = 棚卸しを二層トリガー化し、層2を自動エージェントにする

- Status: Accepted
- Date: 2026-06-28

## Context

昇格は「棚卸し（全 `learn.md` を見直す）」で起こすが、棚卸しにトリガーが無いと
人間の「いつかやる」で永遠に始まらず、`principles/` は空のままになる。
ユーザーは自身を「怠け者」と認識し、過去にメモが続かなかった原因も
**自動で回るバックストップが無かったこと**だと特定した。
（これは「何個で昇格するか」= ADR-0007 とは別の、「いつ棚卸しが起きるか」の問題。）

## Decision

棚卸しのトリガーを二層にする。

- **層1（日和見・任意・複数可）**: プロジェクト完了時などに `learn.md` をざっと見て、
  動かせる原則は自分で `principles/` へ。スキップ可。
- **層2（バックストップ・自動・絶対）**: 毎週、棚卸しエージェント（スケジュール実行の
  Claude）が全 `learn.md` を読み、昇格案を作って承認／却下を求める。

層2は**人間の起動を必要としない（push 型）**ことが必須要件。

## Consequences

- 怠けても棚卸しが回る。メモが続かなかった根本原因（自動バックストップの不在）を解消。
- 起動を人間の記憶力・意志に依存させない。
- 判定基準（ADR-0007）とは独立して機能する。
- Why not:
  - 人間が始める定期棚卸し（毎週日曜に自分でやる等）→ 怠け者には起動自体がスキップされ機能しない（pull 型の罠）。
  - 単一トリガーのみ → 「次の機会でやる」の言い訳になり素通りする。

# ADR-0009: 層2の実行環境 = 当面このノートPCのローカル実行 + Obsidian Sync（将来は自宅サーバ）

- Status: Accepted
- Date: 2026-06-28

## Context

層2（週次の自動棚卸し, [[ADR-0008]]）をどこで走らせるか。
vault には個人情報が含まれるため、GitHub 等の第三者公開リポジトリには置かない方針。
「定期実行 × 常時起動（ノートPC非依存）× データを自分の管理下」の3つは同時に満たせず、
常時起動を求めると自分で管理する always-on マシン（サーバ）が必要になる。

## Decision

当面は **このノートPC 上のローカル実行**（systemd user timer, `Persistent=true`, 毎週月曜 10:00）とし、
データ紛失防止に **Obsidian Sync** を併用する。
エージェントには読み取り系ツールのみ渡し、出力は **提案のみ**（`reviews/` に昇格案ファイルを生成、自動昇格はしない）。

### 将来の展望（always-on 化）

未使用の Windows ミニPC を**自宅サーバ**化し、層2をそこで常時稼働させる:

- ミニPC を常時起動の実行ホストにする（WSL2 か Windows タスクスケジューラで週次実行）。
- vault は **Obsidian Sync / Syncthing** でミニPCにも同期。
- 外部公開はしない。外出先から触りたいときは **Tailscale または WireGuard**（プライベート VPN メッシュ）で接続し、**ssh ポート開放はしない**。

これにより「定期実行 × 常時起動 × データを自分の管理下」の3つを、第三者クラウドに頼らず満たせる。

## Consequences

- privacy を保ちつつ、サーバ構築なしで開始できる。
- ノートPCが寝ていても `Persistent=true` が取りこぼしを次回起動時に拾う（真の常時ではない）。
- 将来の always-on 化の道筋（自宅サーバ + Tailscale/WireGuard）が決まっている。
- Why not:
  - クラウドのマネージド実行 → データが手を離れるため不採用。
  - VPS → 第三者上のため暗号化必須で、今は過剰。
  - 公開サーバ＋ssh ポート開放 → 攻撃面が増え、不要なリスク。

---

# ADR-0010: 分類はフォルダで表さない — projects/ をフラットにし、分類はタグ/MOC（グラフ）で行う

- Status: Accepted
- Date: 2026-06-29

## Context

フォルダ整理は破綻した。症状は3つ:
(1) 階層が深くなりすぎて取り出せない、
(2) ディレクトリを切る粒度がわからない、
(3) 目的によって重複する（同じ主題が複数の親を持つ）。
これらは別々の問題ではなく、1つの根本原因の症状である:
**フォルダ（木構造）に「包含」と「分類」の2つの仕事をさせていた。**
木は1軸・1対多しか表せず、分類は多対多・無限に深くなるため破綻する。
（あるモノに正当な親が2つ以上あるなら、それはもう木ではなくグラフである。）

## Decision

道具を仕事で分ける。

- **包含 → フォルダ（1階層）**: プロジェクトが how/what/why/learn を束ねるのは真の1対多。
  プロジェクトは `projects/` に**フラットに並べる。カテゴリでネストしない。**
- **分類 → タグ／MOC（グラフ）**: frontend・設計・きれいなコード等のカテゴリは
  **絶対にフォルダにしない。** frontmatter のタグと MOC（Map of Content: リンクを束ねる地図ノート, `maps/`）で表す。
  多対多OK・深さなし・重複なし。同じプロジェクトを複数の MOC が指せる。

### 粒度の決定ヒューリスティック

**「達成条件（what）があるか?」** — ある → プロジェクト → フォルダ。ない → カテゴリ → タグ／MOC。

## Consequences

- 深さは常に2以下（`projects/<project>/<file>`）。取り出しはフォルダ堀りでなく検索・タグ・リンク・MOC。
- 重複の悪魔が消える（多対多を許すグラフだから）。
- 凝縮度はプロジェクトフォルダが供給するので、Zettelkasten の「文脈が区切れない」問題も解決。
- 既存の `システム開発の上達/正規表現処理系作成/` はこの規則違反（カテゴリフォルダ）。
  `projects/正規表現処理系作成/` ＋ タグ／MOC へ移行する（別途）。
- Why not: 全部フラットな Zettelkasten → プロジェクトの凝縮度が出ない（過去に試して不適）。

# ADR-0011: 運用層（トラッカー）は別ツールにせず、vault 内のメタdataから生成するビューにする

- Status: Accepted
- Date: 2026-06-29

## Context

HWW（記録の層・過去）、MOC/タグ（地図の層）に加え、第3の層が必要と判明した:
**運用層（前向き）**。「今アクティブか / 抱えすぎてないか / どの頻度・量・いつやるか」を扱う。
これは HWW（過去の記録）とは別概念。別ツール（taskdog 等）に持たせると、
vault とツールで真実が2つに割れる（split-brain）。

## Decision

運用層を **同じ vault 内**に置く。各プロジェクトのメタdata（frontmatter）を**真実の源**とし、
トラッカーはそこから**自動生成するビュー**にする（Dataview 等）。別ツールは使わない。
ただし「促し（nudge）」だけは push 型が必須（ノートは push しない）で、
外部リマインダー／週次エージェントが担う。
詳細な運用ルール（状態・WIP上限・停滞判定・frontmatter schema）は [[ADR-0012]] で定める。

## Consequences

- 真実の源が1つ（プロジェクト自身）。トラッカーは生成物で二重管理なし。
- 倒したばかりの重複の悪魔が「データの重複」として戻るのを防ぐ。
- Why not: 別タスク管理ツール → vault と二重化（split-brain）するため不採用。

---

# ADR-0012: 運用層の運用ルール（状態・WIP上限・停滞判定・frontmatter・ダッシュボード）

- Status: Accepted
- Date: 2026-06-29

## Context

[[ADR-0011]] で運用層を vault 内ビューに決めた。その具体ルールを定める。
ユーザーはプロジェクト管理が苦手なため、最小で破綻しにくい制度にする。

## Decision

- **状態(status)**: `backlog` / `active` / `paused` / `done` / `dropped` の5つ。
  `dropped`（やめた）を正直に使い、死んだプロジェクトが active を圧迫しないようにする。
- **WIP 上限 = 5**: 同時 active は最大5。6つ目を始めるには先に1つを done/paused/dropped にする。
  「始める」より「終える」を優先し、過剰コミットを機械的に防ぐ。
- **停滞判定 = 14日**: active かつ last-touched から14日超で「停滞」。
  週次レビューで 再開／paused／dropped を強制決定する。
- **真実の源 = frontmatter**: 各プロジェクトに `index.md`（表紙ノート）を置き、
  `status / started / last-touched / cadence / dose / tags` を持つ（cadence・dose は稽古系のみ）。
- **ダッシュボード = `dashboard.md`**: Dataview が `projects/` を走査して
  「active一覧・停滞・WIP件数・backlog・全体俯瞰」を自動生成。手で管理しない。
- **促し(nudge) = push**: ダッシュボードは pull なので、停滞・WIP超過の通知は週次エージェント、
  日次の稽古は外部リマインダーが担う。

## Consequences

- 「多すぎ?」は WIP 上限で、「やってるつもり?」は停滞判定で、機械的に答えが出る。
- Dataview プラグインに依存する（嫌なら週次エージェントが静的ダッシュボードを生成する代替あり）。
- 各プロジェクトは how/what/why/learn の4ファイル＋ `index.md` の5点になる。
- Why not: しきい値の最適値を議論し続ける → 続けることが最優先。標準値(5/14)で走り、痛みが出たら調整。

# ADR-0013: 捕捉バッファ inbox.md を許容する（捕捉と整理を分離）

- Status: Accepted
- Date: 2026-06-29

## Context

疲れているとき、how/what/why/learn のルールを全部守りながら書くのは難しい。
ルールを課すと、疲れた状態の思いつきが捕捉されず失われる。
一方で「tmp.md」的な自由バッファは、処理されず腐る危険がある（ダンプ化）。
GTD（Inbox）・Zettelkasten（fleeting note）・デイリーノートは、いずれも
「捕捉（ルールなし）と整理（後でルールあり）を分け、バッファは処理義務とセット」で解いている。

## Decision

vault ルートに `inbox.md` を置く。摩擦ゼロの捕捉先で、how/what/why/learn を考えずに
時刻付きで1行放り込む。後で project の how/learn へ振り分ける／新 project を起こす／
principle にする／**捨てる**。捨てるのは正解（解読不能な走り書きはゴミ）。
inbox は溜める場所でなく通り抜ける場所。**7日**超の未処理は処理 or 削除。
処理の促しは月曜のポートフォリオ点検に同梱（push）。
「疲れたらやらない」は採らない — システムは最悪の状態でも回るべき。

## Consequences

- 疲れた自分のための安全弁を、腐らせない保証（push）付きで持てる。
- 捕捉（inbox）→ 整理（HWW）→ 昇格（principles）という流れが一本化。
- Why not:
  - ルールを常に強制 → 疲れた状態の思いつきを失う。
  - 期限なしの自由 tmp → ダンプ化して腐る。
  - 疲れたらやらない → 損失。脆い設計。

# ADR-0014: カンバン型タスク管理ツールの導入を見送る（要件不足・運用後に再考）

- Status: Rejected（条件付きで将来再考）
- Date: 2026-06-30

## Context

運用層（[[ADR-0011]] / [[ADR-0012]]）を vault 内ビューに決めた後、
カンバン型タスク管理の導入を再検討した。検討で判明したこと:

- 欲しかったものの大半は「描画(A)」であり「独立ツール(B)」ではなかった。
- 「一箇所で捕捉」は inbox.md（[[ADR-0013]]）で既に解決済み。未解決なのは
  「俯瞰＋ドラッグで状態を動かす常設ボード」だけ。
- その実体は「task という entity が現状どこにも住所を持たない」ことの裏返しだった
  （how=過去ログにタスクを詰めようとしていた）。
- entity を (a) 各プロジェクト内 `task.md` に置けば split-brain は避けられる、
  までは設計できた。

## Decision

現時点では**カンバンツールを導入しない**。理由は、それを正当化する運用上の痛みが
まだ実在しない（要件が弱い）から。先に既存の運用層（dashboard / 停滞判定 / inbox）を
回し、痛みが具体化してから再考する。

## Consequences

- 弱い動機で (b) 的な二重化に踏み込むのを回避（[[ADR-0011]] の split-brain 警告を尊重）。
- 再考の条件（これが起きたら見直す）:
  - 「思いついたタスクをプロジェクトまで移動して書くのが面倒」が実際に捕捉ロスを生む。
  - dashboard の一覧では「次に何をやるか」が判断できない場面が反復する。
- 再考時に持ち越す設計メモ:
  - entity を導入するなら (a) 各プロジェクト `task.md` を真実の源にする。
  - 自作ツールは状態を自前 json に持たず、必ず task.md に書き戻す（さもなくば (b)）。
  - 反復タスク（稽古）は done 後のリセットと「サボり検出」が一回性と別物。
    安易に同じボードに載せない（案X: ボードは一回性専用が筋）。
- 未決事項: inbox を vault ルート1個（[[ADR-0013]]）とするか、プロジェクト別に置くかで
  ユーザーの想定と食い違いがある。別途決着が必要。
- Why not（今あえて入れない理由）: 痛みの実在しない最適化は、続けることを最優先する
  本システムの方針（[[ADR-0012]]）に反するため。

# ADR-0015: how を廃止し log を新設する（vault はメタ層・成果物は持たない）

- Status: Accepted
- Date: 2026-06-30
- Supersedes: [[ADR-0001]]（隔離原則は継承）, [[ADR-0002]]（learn は継承）

## Context

how の定義が「過去にやった事実のログ」で運用されてきたが、ソフト開発の比喩と矛盾する。
TDD では **what = テスト**を先に書き、それを満たす **how = 確実に動くコード**へ落とす。
つまり how は本来「過去ログ」ではなく「**実行される成果物（コード）**」を指す。

ここで実行者(executor)の非対称が効く:

- **コード**: 機械は自分で動けないので、命令＝how（コード）を**書く必要がある**。how が実行される。
- **人間**: 実行するのは**身体・行動そのもの**で、書かれた成果物ではない。
  「実行されるための how」を書く意味がない。

さらに、ソフト開発でも**コード（how）は ADR フォルダに置かない**。コードはコードの住処にある。
ADR/test/lint（why/what/learn）だけがメタ層。**vault は元々メタ層**なのだから、
how（実物）がそこに無いのは当然だった。how は最初から vault の住人ではない。

how は1スロットで3役を兼任していた: (1)動く成果物、(2)実行の記録、(3)教訓。
(3)は既に learn へ分離済み（[[ADR-0002]]/[[ADR-0003]]）。残る (1)(2) の整理が本決定。

## Decision

- **`how.md` を廃止する。** 「動く how（実物）」は固有の住処に在る:
  稽古系 → **身体**、制作系 → **コードリポジトリ**、文章系 → **原稿**。vault はそれを持たない。
- **`log.md` を新設する。** how が兼ねていた「実行の記録（何をやった・結果どうだった）」を担う。
  生の事実が原料、learn（教訓）は蒸留物。両者は別バケツ（[[ADR-0003]] の混濁回避と一貫）。
- **`what.md` に「達成条件 ＋ テストケース」を持たせる。** 「何を達成したら基準が満たされたと
  言えるか」を具体的な受け入れ条件（テスト相当）として書く。
- バケツは **how/what/why/learn → what/why/learn/log** に再編。数（4つ）と「認識タイプで
  物理隔離する」原則（[[ADR-0001]]）は維持。how を log に置換しただけ。

## Consequences

- 「人間に how（実行される成果物）を書かせる」不自然さが消える。vault＝メタ層が徹底される。
- 制作系の成果物（コード等）を vault に入れるか native な住処に置くかは**別途未決**（notDecided 参照）。
- 既存ドキュメント（README / glossary / rule / template / 各 project の `how.md`）は
  how→log 前提に書き換えが必要（実装タスク・別途）。
- learn の独立は維持（log は learn ではない）。
- Why not:
  - how を「過去ログ」のまま残す → ソフト比喩と矛盾し、定義が曖昧なまま運用が揺れる。
  - 実行記録を learn に混ぜる → 原料と蒸留物の混濁（[[ADR-0003]] が回避したもの）。
  - 4バケツを3バケツに減らす（log も作らない）→ 実行記録の住所が消え、learn か what へ漏れる。

# ADR-0016: jotatsu 核は what/why/learn（現在・確定した情報）だけ。未確定なものは別ディレクトリへ

- Status: Accepted
- Date: 2026-07-05
- 関連: [[ADR-0015]]（how→log の流れ）

## Context

jotatsu の責務を広げすぎていた。what/why/learn は「**現在かつ確定した情報**」だが、
実行計画やタスク管理は未確定・時限的で性質が違う。同じ器に混ぜると確定と未確定が混濁する。

判定の軸（時間 × 確定度）:

| 対象 | 時間 | 確定度 | jotatsu 核 |
|---|---|---|---|
| what（達成条件） | 現在 | 確定 | **入れる** |
| why（判断） | 過去だが現在向き | 確定 | **入れる** |
| learn（知見） | 現在向き | 蒸留済み | **入れる** |
| 実行計画 / roadmap | 未来 | 未確定・時限的 | 別ディレクトリ |
| タスク管理 | 現在/未来の運用 | 未確定・時限的 | 別ディレクトリ |
| log（実行記録） | 過去 | 生・未蒸留 | 別ディレクトリ |
| inbox（捕捉） | 未確定 | 未蒸留 | 別ディレクトリ |

why が過去でも入るのは、確定した再利用知だから。軸の本質は時間より**確定度**。
核に残るのは「確定・蒸留された現在向きの再利用知」だけ。

## Decision

- jotatsu 内に別ディレクトリを切り、そこに入れるのは **what / why / learn だけ**。理由は
  この3つが「現在かつ確定した情報」だから。中核手法の名を **howWhatWhyLearn → whatWhyLearn** に改称。
- **核以外（実行計画・タスク管理・log・inbox）は核外の別ディレクトリで管理**する。核に残すのは
  「確定・蒸留された現在向きの再利用知」だけ、で線を引く。
- 物理分離（dir か repo か）は責務分離を保証しない。保証するのは書く規律。だからサブディレクトリで
  分ければ足り、別リポジトリ化は必須ではない。

## Consequences

- 核が「現在・確定」に純化され、確定と未確定の混濁が消える。
- 実行計画・タスクの痛みは消えないので、分けた先のディレクトリで別途解消する。
- README / rule / glossary / template 等の how・howWhatWhyLearn 表記は書き換えが必要（別途）。

# ADR-0017: プロジェクトの形（what/why/learn＋生 note.md）、実態は外部＋link、learn は craft の知見

- Status: Accepted
- Date: 2026-07-05
- 関連: [[ADR-0016]]（核純化）, [[ADR-0014]]（タスクツール見送り）, [[ADR-0002]]（learn 追加）

## Context

実行計画（roadmap）をテンプレ化できるか検討したが、中身は未知・プロジェクト依存・想定外で
定式化できないしすべきでない。また「疲れていても最初から認識タイプを切り分けて書くのは無理」。
分離を"書く時"に強制すると書けなくなる。加えて learn の守備範囲が曖昧で、ドメイン技術知識まで
learn に流れ込む懸念が出た。

## Decision

実行計画は別ディレクトリで管理しない。
1. マイルストーンや定期的な振り返りといった仕組みはruleで定式化する
2. なにをやったか、思いつき、得られた知見は自由記述で書く。(note.md)

**プロジェクトの構成:**
  ```
  projects/<p>/
  ├── index.md   表紙・frontmatter（status / cadence・dose / tags / 実態への link）
  ├── what.md    核・確定
  ├── why.md     核・確定
  ├── learn.md   核・確定（蒸留の受け皿）
  └── note.md    核外・生の working note（log＋「今こうしよう」＋知見の種を混ぜて書く）
  ```

note.mdに一時的に書きなぐり、あとからlearnに分ける。タスクは note に入れず、外部ツールを使う。

プロジェクトの実態（動く実物・ドメインの技術知識）は別の箇所（GitHub・zettelkasten 等）で管理し、link を貼る

learn.mdには進め方の知見を入れる。
ex. 「GOでHTTP Serverを作ったときの知見」 -> Zettelkasten
ex.  HTTPサーバを作るときHTTPの仕様を先に見たほうが良かった -> learn.md


## Consequences

- 「始めやすい」形になり、実プロジェクトでの運用を開始できる
- note.md が一時的にlog、learn、思いつきをひきうける
- learn が craft に純化され、ドメイン技術知識で膨れない。zettelkasten との役割分担が明確化
- Why not:
  - 実行計画をテンプレ化 → 未知・想定外を定式化する無理筋。中身が腐る。
  - 書く時から log/roadmap/learn を分離 → 疲れた自分が書けず捕捉ロス。
  - learn にドメイン知識も入れる → craft の蒸留ラインが技術メモで埋もれ、昇格が機能しない。
  - タスクを自作 → 痛みが実在しないうちの最適化（[[ADR-0014]]）。まず外部で回す。
